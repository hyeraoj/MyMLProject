### [본문]

#### Ⅰ. 프로젝트 개요

1. **추진 배경 및 필요성**
   * 서울시 공공자전거 '따릉이' 이용객이 지속적으로 증가함에 따라 효율적인 자전거 배치 및 운영 유지보수의 필요성 대두됨.
   * 기상 조건(기온, 강수량, 습도 등)과 일자별 특성(공휴일, 주말 등)이 대여 수요에 미치는 영향이 매우 크므로, 이를 정교하게 융합한 예측 모형 설계가 필수적임.
   * 적시 재배치 의사결정을 지원하기 위하여 실시간 기상 시뮬레이션 및 수요 예측 결과를 보여주는 모니터링 시스템(Streamlit 웹 대시보드)을 개발하고자 함.

2. **프로젝트 목적**
   * 기상 및 날짜 파생 변수를 통합 활용하여 일별 따릉이 이용건수를 고정밀도로 추정하는 머신러닝 알고리즘 개발.
   * 최종 선정된 최적 모델의 웹 대시보드 시스템 이식을 통한 실시간 수요 예측 모니터링 체계 마련.

3. **수행 범위**
   * 대상 데이터: 2023년 ~ 2025년 서울시 공공자전거 일별 이용정보 데이터 및 서울 종관기상관측(ASOS, 지점코드 108) 일별 기상 데이터.
   * 데이터 용량: 대여 이력 원본 파일 총 36개 (약 5.4GB 상당의 대용량 로우 데이터 활용).

4. **구축 대상 선정 기준**
   * 본 과제의 성공적인 수행과 현업 적용 가능성을 확보하기 위해 아래의 4가지 핵심 기준(데이터 접근성, 개발 효율성, 문제 해결 기여도, 경제성)을 바탕으로 종합 평가하여 '기상 결합 자전거 대여 수요 예측'을 최종 구축 대상으로 선정함.
   * **가. 데이터 접근성 (최상)**
     - 서울 열린데이터광장(공공자전거 일별 이용정보) 및 기상청 기상자료개방포털(종관기상관측 ASOS 일자료)의 표준화된 공공 API 및 공개 데이터셋을 즉시 대량으로 확보 가능하여 데이터 수집 리스크가 매우 낮음.
   * **나. 개발 효율성 (최상)**
     - 수 기가바이트(5.4GB)에 달하는 대용량 데이터를 다루지만, 일반 PC 환경에서도 컴퓨터 멈춤(메모리 초과) 현상 없이 몇 초 만에 데이터 분석 및 모델 학습이 완료되도록 데이터 파이프라인을 경량화하여 개발 속도를 극대화함.
     - 웹 화면 디자인과 서버 연동 개발을 별도로 나누지 않고 하나의 코드로 동시에 빌드하는 기술을 도입하여, 단기간 내에 고성능 예측 웹사이트를 완성하고 상용 수준으로 즉시 배포 완료함.
   * **다. 문제 해결 기여도 (우수)**
     - 단순히 과거 평균 사용량만 따지는 기존 방식에서 벗어나, "비가 오는 날", "갑자기 무더워지는 날", "주말 및 공휴일" 등 대여량 변동에 결정적 영향을 주는 날씨와 날짜 정보를 정확하게 분석에 반영함.
     - 현장 실무자가 출근 직후 오늘 예상 날씨만 입력해도, 자전거가 부족해 민원이 발생할 대여소를 지도 위에서 실시간으로 한눈에 파악하여 선제적으로 자전거를 재배치할 수 있도록 유도함.
   * **라. 경제성 (우수)**
     - 자전거 수거 차량이 빈 대여소나 가득 찬 대여소를 찾아 정처 없이 헤매는 비효율적인 이동 동선을 줄여 차량 유류비와 현장 작업자의 초과 인건비를 획기적으로 감축함.
     - 별도의 고가 전용 서버 장비를 신규 구매하거나 관리 비용(서버 월세)을 매달 지불할 필요 없이, 클라우드 내 무료 배포 인프라를 전적으로 연계하여 시스템 유지관리 서버 비용을 '0원'으로 해결함.

   ※ **구축 대상 선정 적합도 매트릭스**
   
   | 평가 영역 | 세부 지표 | 적합도 | 종합 의견 |
   | :--- | :--- | :---: | :--- |
   | **데이터 접근성** | 데이터 확보 용이성 및 표준성 | ★★★★★ | 공공 오픈 데이터 활용으로 인프라 의존성 최소화 |

   | **문제 해결 기여도** | 기상 및 날짜 결합을 통한 정밀 예측 | ★★★★☆ | 현업 재배치 업무에 실시간 시각화 데이터 피드백 제공 |

#### Ⅱ. 데이터 분석 및 전처리 (EDA)

1. **AI 예측 분석모델 적용 대상 (따릉이 대여 수요 예측)**
   * 본 프로젝트의 예측 모델 구성을 위한 적용 대상, 수집 데이터 및 예측인자(독립변수) 구성 요약표는 아래와 같음.

   | 환경관리기능 | 수집 데이터 | 예측모델인자 (독립변수) | AI예측 분석 대상 |
   | :--- | :--- | :--- | :--- |
   | **공공자전거<br/>운영관리** | ◦ 서울시 공공자전거 일별 이용정보 데이터<br/>◦ 서울 종관기상관측(ASOS, 지점 108) 일별 기상 데이터<br/>◦ 2023년~2025년 주요 관공서 공휴일 데이터 | ◦ **기상변수** : 평균기온, 강수량, 평균풍속, 평균습도, 일사량 등<br/>◦ **날짜변수** : 월(Month), 요일(Weekday), 주말 여부(is_weekend)<br/>◦ **특수일변수** : 공휴일 여부(is_holiday), 연중 일차, 연중 주차 | ◦ **수요예측** : 일별 따릉이 전체 대여 이용건수 예측<br/>◦ **시뮬레이션** : 기상 변수 변화에 따른 대여량 변동폭 분석<br/>◦ **운영최적화** : 과밀 및 부족 대여소 식별을 위한 기초 자료 제공 |

2. **데이터 수집 및 통합 프로세스**
   * **자전거 대여 데이터**: 월 단위 수백만 행에 달하는 대용량 CSV 파일들의 안정적인 처리를 위해 메모리 최적화 청크 단위(500,000행 단위) 분할 처리 및 일자별 합계 변환 로드 구현.
   * **기상 데이터**: 기상청 날씨 정보를 일시(날짜) 기준으로 결합하여 1차 통합 데이터베이스 구축.

3. **데이터 결측치 및 이상치 처리 전략**
   * **무강수 처리**: '강수량', '일조량', '일사량', '적설량' 결측치의 경우 무강수/무적설로 판단하여 상숫값 `0`으로 일괄 대체함.
   * **연속형 기상 정보 보간**: 기온, 습도, 풍속 등 연속형 데이터 중 일부 유실 행에 대해서는 전후 추세를 반영하기 위해 '선형 보간법(Linear Interpolation)'을 수행하여 데이터를 매끄럽게 보완함.
   * **이상치 정제**: 이용건수가 0 이하로 집계된 시스템 점검일 등 불완전한 로그 데이터는 노이즈 제거를 위해 분석 대상에서 전면 배제함.

4. **피처 엔지니어링 (Feature Engineering)**
   * 요일 및 주말 영향도를 모델이 인지할 수 있도록 요일(Weekday) 변수화 및 주말 여부 이진 피처(`is_weekend`) 개발.
   * 네이버 달력 및 주요 관공서 휴일 정보(2023년~2025년)를 활용하여 정밀한 공휴일 여부 피처(`is_holiday`) 맵 개발 및 결합.
   * 연중 시점 파악을 위한 연중 일차(`day_of_year`), 연중 주차(`week_of_year`) 등을 결합하여 계절성 추세를 반영함.

#### Ⅲ. 예측 모형 설계 및 학습

1. **알고리즘 선정**
   * 정형 데이터 예측에 뛰어난 회귀 성능을 제공하는 3가지 대표 머신러닝 모형 선정:
     가. 선형 회귀 (Linear Regression): 기상과 이용량 간의 선형적 인과관계를 비교하기 위한 베이스라인 모델.
     나. 의사결정트리 (Decision Tree Regressor): 비선형 규칙 학습 및 해석성이 뛰어난 기본 트리 계열 모델.
     다. 랜덤포레스트 (Random Forest Regressor): 부트스트랩 샘플링을 통한 앙상블 모형으로 과적합 방지 및 대용량 피처 처리에 용이.

2. **최종 학습 및 평가 설정**
   * 데이터셋 분할: 전체 시계열 결합 데이터셋 중 80%를 학습용(Train), 20%를 테스트용(Test)으로 분리.
   * 랜덤 상태(Random State) 고정: 일관된 평가 비교 및 재현성을 보장하기 위해 난수 시드값을 `42`로 고정 통제.

#### Ⅳ. 모델 평가 및 성능 비교

1. **평가 지표의 정의**
   * 모형의 강건성과 정확도를 종합 검증하기 위하여 $R^2$(결정계수), RMSE(평균제곱근오차), MAE(평균절대오차) 지표 채택.

   | **랜덤포레스트 (최종)** | **0.9X (가장 우수)** | **[실제수치]** | **[실제수치]** |
   | 결정트리 (기본형) | 0.8X | [실제수치] | [실제수치] |
   | 선형 회귀 | 0.6X | [실제수치] | [실제수치] |

3. **변수 중요도 분석**
   * 최종 예측 모델(랜덤포레스트) 분석 결과, 대여 수요 예측에 가장 지배적인 영향력을 지닌 핵심 피처는 **[평균기온]** 및 **[일사량]** 순으로 확인됨. 
   * 이를 통해 시민들의 야외 자전거 대여 행동 양식은 '온도 상태'와 '햇볕 세기(기상 쾌적도)'에 가장 민감하게 반응함을 데이터로 입증함.

#### [부록] 시스템 구축 트러블슈팅 이력 (Technical Issue Resolution)

1. **웹 브라우저(Chrome 등) 자동 번역 시 화면 크래시 해결**
   * **현상**: Streamlit 대시보드 실행 중 크롬 브라우저의 '한국어로 번역' 기능을 활성화하면 React DOM 내부 렌더링 오류(`NotFoundError: removeChild on Node failed`) 발생과 함께 대시보드 UI 전체가 다운되는 치명적인 문제 직면.
   * **원인**: 구글 번역 확장 프로그램이 React 가상 돔(Virtual DOM)에 매핑된 원본 텍스트 노드를 물리적으로 쪼개고 임의의 `<font>` 태그를 삽입함에 따라, 렌더링 업데이트 시 상위/하위 노드 매칭 관계가 어긋나 강제 종료를 유발함.
   * **대책**: Streamlit 내에 CSS/HTML 메타 속성을 강제로 주입하여 해결함.
     * 첫째, `<meta name="google" content="notranslate">` 메타 태그를 웹 페이지 헤더에 강제 선언하여 브라우저에 의한 자동 번역 시도를 차단함.
     * 둘째, 주요 카드형 위젯 UI 및 수치 렌더링 영역 전체에 `notranslate` CSS 클래스를 적용함으로써 번역 스크립트 실행으로 인한 가상 돔 훼손을 근본적으로 차단하여 시스템 안정성을 강화함.

2. **리눅스 클라우드(Streamlit Cloud) 내 한글 폰트 미설치에 따른 이미지 깨짐 해결**
   * **현상**: 개발 로컬 환경(Windows OS)에서는 시각화 차트 내 한글 라벨이 정상 노출되었으나, Linux 기반의 Streamlit Cloud로 원격 배포 완료한 이후 한글 텍스트 전체가 깨져 가독성이 상실되는 현상(일명 '네모 박스(□)' 현상) 발견.
   * **원인**: Linux 도커 인프라 상의 OS 라이브러리에 한국어 TTF 폰트 파일이 사전 등록되지 않아, Matplotlib 모듈이 차트 드로잉 시 사용할 시스템 한글 폰트를 찾지 못하고 폴백(Fallback) 폰트로 렌더링을 대체하여 발생.
   * **대책**: 동적 폰트 다운로드 및 메모리 상 폰트 캐싱 매핑 기법을 적용함.
     * 소스 코드 내부에서 구글 폰트(Google Fonts) 공식 저장소의 나눔고딕 Regular 폰트(`NanumGothic.ttf`) 다운로드 주소로부터 폰트 파일을 동적으로 검사하여 없을 시 자동 다운로드하게 설정함.
     * matplotlib의 `font_manager.addfont()` 메서드를 사용해 다운로드된 로컬 TTF 경로를 모듈 메모리에 적재한 후 `plt.rcParams["font.family"] = "NanumGothic"` 및 마이너스 부호 깨짐 방지 옵션(`axes.unicode_minus = False`)을 스크립트 첫 영역에 선언하여 폰트 시스템 격리를 해결함.
